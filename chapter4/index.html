Stage 01 — Planning & Requirements
Stage 01 · Planning

Build the PRD with AI

📄 Deliverable: Product Requirements Doc
Goal: Turn a vague product idea into a structured, AI-ready PRD that defines what you're building, who it's for, what features it needs, and what success looks like.
To-Do Checklist
1.1 Write a 1-paragraph product brief — describe the product, target user, and core problem in plain language.
1.2 Prompt AI to expand into a full PRD — problem statement, user personas, user stories, acceptance criteria.
1.3 Define priority levels — mark each feature as P0 (must), P1 (should), or P2 (nice-to-have).
1.4 Define out-of-scope items — explicitly tell AI what NOT to build to keep scope clear.
1.5 Human review & sign-off — read every line. Edit until it reflects your real intent. Save as PRD.md.
Expected Output
PRD.md (Structured Requirements)
Defined Scope (P0/P1/P2 features)
Starter Prompt Template
PRD GENERATOR PROMPT
# ROLE
You are a senior product manager.
 
# TASK
Based on this brief, write a complete PRD:
[BRIEF] {paste your 1-paragraph idea here}
 
# OUTPUT FORMAT
Include: Problem Statement, Target Users,
User Stories (As a... I want... So that...),
Acceptance Criteria, Priority (P0/P1/P2),
Out of Scope, Success Metrics.
 
# FORMAT: Markdown. Be structured and specific.
Context to Feed the AI
📝 Your 1-para product brief 👤 Target user description 🔍 Competitor references 📊 Any existing research
✦ Pro tip: After the AI generates the PRD, ask it: "What edge cases am I missing? What assumptions am I making?" — this catches gaps before you write a single line of code.
Stage 02 — Architecture & Design
Stage 02 · Architecture & Design

Architect & Design with AI

🏗️ Deliverable: System Design Doc
Goal: Transform your PRD into a concrete technical blueprint — tech stack decisions, database schema (ERD), API structure, and infrastructure plan — all reviewed by you before a single file is created.
To-Do Checklist
2.1 Paste your PRD and ask AI to recommend a tech stack with clear reasoning for each choice.
2.2 Generate the ERD — ask AI to produce a database schema (entity-relationship diagram) from your features.
2.3 Generate the API contract — endpoints, request/response shapes, auth method (OpenAPI spec format).
2.4 Map the system layers — Frontend → Backend → AI API → Database → Storage → Infra.
2.5 Human review & finalize — edit, question AI's choices, then save as DESIGN.md.
Expected Output
DESIGN.md (Tech Stack + Schema)
API Contract (OpenAPI/JSON Spec)
Starter Prompt Template
DESIGN PROMPT
# ROLE
You are a senior software architect.
 
# CONTEXT
[PRD] {paste full PRD.md here}
 
# TASK
Design the full system design:
1. Recommend tech stack (with reasons)
2. Database schema + ERD (Markdown table)
3. API endpoints list (REST or GraphQL)
4. System layers diagram (text ASCII)
5. Identify top 3 technical risks
 
# Prioritize: simplicity over complexity.
Context to Feed the AI
📄 Full PRD.md 🧑‍💻 Team skill level ☁️ Deployment target 💰 Budget constraints
✦ Pro tip: Ask AI: "What would break at 10,000 users? What would you simplify if we only had 2 weeks?" Forces it to prioritize ruthlessly — exactly what a good designer does.
Stage 03 — Implementation (Vibe Coding)
Stage 03 · Implementation

Build Layer by Layer

⚡ Deliverable: Working Codebase
Goal: Transform design into functional code in structured layers, always with the full context (PRD + Design) loaded, and always checkpointing before moving to the next task.
To-Do Checklist (Layer Order)
3.1 Load full context first — paste PRD + Design Doc into the AI before writing any code.
3.2 Layer 1 — Frontend: Setup → UI Components → API connection. ✅ Checkpoint: UI renders correctly.
3.3 Layer 2 — Backend: Server setup → DB schema → API endpoints → Auth. ✅ Checkpoint: API responds.
3.4 Layer 3 — AI Integration: Connect Gemini → implement AI feature logic → test responses. ✅ Checkpoint: E2E works.
3.5 One task at a time — never queue multiple tasks. Complete → checkpoint → commit → proceed.
Expected Output
Functional Codebase (Frontend + Backend)
Integrated AI Reasoning Pipeline
Context-Loading Prompt (use at session start)
IMPLEMENTATION CONTEXT PROMPT
# PROJECT CONTEXT — Read before doing anything.
 
[PRD] {full PRD.md}
[DESIGN] {full DESIGN.md}
[CODEBASE] {current file tree}
 
# CURRENT TASK
Implement: "{one specific task}"
 
# RULES
— Only implement what's in this task.
- Follow the design system strictly.
— Tell me when done + what to test next.
Context to Feed the AI Every Session
📄 PRD.md 🏗️ DESIGN.md 📁 File tree 🐛 Error logs ✅ Last checkpoint status
⚠️ Golden Rule: Never give the agent more than one task at a time. Parallel tasks → file conflicts → untraceable bugs. One task, one checkpoint, one commit.
Stage 04 — Testing
Stage 04 · Testing

Test with AI, Verify as Human

🧪 Deliverable: Test Suite + Bug Report
Goal: Use AI to generate comprehensive tests from your code and PRD acceptance criteria — then you (the human) run them, evaluate the results, and sign off before shipping.
To-Do Checklist
4.1 Unit tests: Paste each function → ask AI to write tests covering happy path, edge cases, and failure modes.
4.2 Integration tests: Feed the API contract → ask AI to generate end-to-end test scripts for each endpoint.
4.3 AI feature tests: Manually test AI outputs for accuracy, hallucination, and edge case handling — this requires HUMAN judgment.
4.4 Bug report generation: Paste failing test output → ask AI to identify root cause and suggest a fix.
4.5 UAT checklist: Ask AI to generate a user acceptance testing checklist from PRD acceptance criteria.
Expected Output
tests/ (Unit & E2E Test Suite)
bug_report.md (Verification Log)
Test Generation Prompt Template
TEST GENERATION PROMPT
# ROLE: Senior QA Engineer
 
# CONTEXT
[FUNCTION] {paste the function/component}
[ACCEPTANCE] {paste acceptance criteria from PRD}
 
# TASK
Write complete test cases covering:
— Happy path (expected inputs)
— Edge cases (boundary values)
— Failure modes (invalid inputs)
— Security edge cases
 
# Use: {testing framework, e.g. Jest / Pytest}
Context to Feed the AI
📋 Acceptance criteria 🔧 The function/endpoint 🐛 Failing test output 📊 Test framework name
✦ HITL Rule: AI-generated tests must be run and verified by a human. AI output quality evaluation — especially for LLM features — cannot be automated. Your judgment is the final checkpoint.
Stage 05 — Review & Iterate
Stage 05 · Review & Iterate

Review, Refine, Ship

🔍 Deliverable: Refined, Production-Ready Product
Goal: Use AI as a code reviewer, performance auditor, and security checker — then make the final calls yourself. Iterate in tight loops until the product meets your PRD's success metrics.
To-Do Checklist
5.1 Code review: Paste code → ask AI to review for readability, maintainability, and anti-patterns.
5.2 Security audit: Feed backend routes and auth logic → ask AI to identify injection, CSRF, and auth vulnerabilities.
5.3 Performance check: Ask AI to identify N+1 queries, unnecessary re-renders, and unoptimized API calls.
5.4 PRD alignment check: Paste PRD + current features → ask AI: "What's missing? What doesn't match the spec?"
5.5 Iteration loop: Fix → test → review → commit. Repeat until all P0 criteria are met and approved by you.
Expected Output
Production-Ready Application
Final Performance & Security Sign-off
Code Review Prompt Template
CODE REVIEW PROMPT
# ROLE: Principal Engineer doing code review
 
# CONTEXT
[CODE] {paste file or function}
[PRD] {paste relevant section}
 
# REVIEW FOR
1. Correctness — does it do what the PRD says?
2. Security — any vulnerabilities?
3. Performance — any bottlenecks?
4. Readability — is this maintainable?
5. Missing cases — what could break?
 
# Output: Prioritized list of issues (P0/P1/P2)
Context to Feed the AI
💻 Code file(s) 📄 Relevant PRD section 📊 Performance metrics 🔒 Auth/security context
✦ Remember: AI makes the suggestions — you make the decisions. Review every recommendation critically. AI can miss business context, user nuance, and organizational constraints.
Best Practices for AI-Driven Development

8 Rules Every AI-Driven Developer Must Follow

Best Practices

01
One task at a time — always
Never queue multiple tasks for the agent. One task → checkpoint → commit → next task. Sequential is faster and safer.
02
Load full context every session
Start every new session by pasting your PRD, design doc, and current file tree before any task.
03
Commit before every agent task
If the agent breaks something, you need a clean rollback. git commit is your safety net — use it religiously.
04
Always review before merging
AI is fast, not infallible. Read every line it writes. You are the last line of defense between AI output and production.
05
Store prompts in /docs or /ai
Your PRD, design doc, and key prompts are source-of-truth documents. Commit them to the repo alongside the code.
06
Reset context when agent gets stuck
A confused agent produces garbage. If output quality drops, start a fresh session with a clean, focused prompt.
07
Test with messy real-world inputs
AI outputs are unpredictable with edge case inputs. Always test with unexpected, malformed data before shipping.
08
You are the architect — AI is the engineer
Set goals. Define constraints. Make decisions. Review output. AI generates; you direct, approve, and take responsibility.
